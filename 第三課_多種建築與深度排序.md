## Step 2:核心觀念一 ——「用資料描述遊戲內容」

請看程式中段的 `CATALOG`(建築型錄):

```gdscript
var CATALOG := {
    "house": { "name":"木屋", "cost_wood":10, "produce":"wood", "amount":2, "height":30.0, ... },
    "farm":  { "name":"農田", "cost_wood":8,  "produce":"food", "amount":2, "height":8.0,  ... },
    "mansion": { ... },
}
```

注意:我們**沒有為每種建築寫一段不同的程式**。所有建築共用同一套蓋房、產出、繪圖邏輯,差別只在這份「資料表」裡的數字與顏色。

這是專業遊戲開發最重要的觀念之一,叫 **data-driven(資料驅動)**:

> 程式負責「怎麼做」,資料負責「做什麼」。

想加第四種建築(例如礦場)?你**不用改任何邏輯**,只要在 `CATALOG` 多加一筆、在 `order` 陣列加上它的名字就好。你的城市建設遊戲未來會有幾十種建築,全靠這個模式才不會失控。

## Step 3:核心觀念二 —— 深度排序(2.5D 的靈魂)

這是這一課最重要的技術。先做個實驗感受問題:

在等距視角裡,如果你把一棟高樓蓋在「後面」、矮房蓋在「前面」,**前面的矮房應該擋住後面高樓的底部**。但電腦怎麼知道誰該畫在上面?

答案:**畫圖有先後順序,後畫的蓋住先畫的。** 所以我們必須「由後往前」畫——先畫最遠的,再畫近的。

關鍵在這段:

```gdscript
var cells := buildings.keys()
cells.sort_custom(func(a, b): return (a.x + a.y) < (b.x + b.y))
for c in cells:
    _draw_building(c.x, c.y, buildings[c])
```

`col + row` 的值越大,代表這格越靠近螢幕下方(越「前面」)。我們把所有建築依這個值由小到大排序,再依序畫——遠的先畫、近的後畫蓋上去,遮擋就正確了。

**這就是 Godot TileMap 的「Y-sort」在背後做的事**。我們現在手動做一遍,等之後換成圖片素材,就能直接開啟 Godot 內建的 Y-sort 功能,原理完全一樣。

> 動手驗證:把那行 `sort_custom` 整行刪掉(或註解掉),再蓋幾棟高低不同的房子互相重疊,你會看到遮擋變得很亂。看完記得加回來。

## Step 4:動手玩設計

1. 在 `CATALOG` 裡複製一份,做出第四種建築「礦場」:成本高、產出大。記得到 `order := ["house","farm","mansion"]` 加上 `"mine"`,並在 `_unhandled_input` 加上按 `4` 選它(仿照現有的 KEY_1~KEY_3)。
2. 調整三種建築的成本與產出,設計一個「合理的經濟」:讓農田便宜但產少、大宅貴但產多,玩家要權衡。
3. 把大宅的 `height` 改成 `90.0`,蓋一片高樓,觀察深度排序是否仍正確。

## 你這一課學到的

- **資料驅動設計**:用一張 `CATALOG` 資料表描述所有建築,邏輯共用。這是讓遊戲能「長大」的關鍵。
- **深度排序**:等距/2.5D 遊戲靠「由後往前畫」處理遮擋,排序鍵是 `col + row`。
- 用 `sort_custom` 與 lambda(匿名函式)排序。
- 字典裡存複雜資料(每種建築是一個小字典)。
